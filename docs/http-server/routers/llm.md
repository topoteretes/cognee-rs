# Router: llm

The `/api/v1/llm` router exposes two thin LLM-utility endpoints used by the cognee frontend's "Schema Builder" UI: `POST /custom-prompt` synthesizes a custom extraction prompt from a JSON-Schema-style graph model, and `POST /infer-schema` proposes a graph model from sample text. Both delegate directly to `cognee-llm` adapters via two prompt templates in `cognee-cognify`'s prompt library; neither touches the relational DB, vector DB, or graph DB. The endpoints exist purely to let the frontend bootstrap a `graph_model` argument for `/api/v1/cognify`.

Companion docs: [../plan.md](../plan.md), [../architecture.md](../architecture.md), [../auth.md](../auth.md), [../observability.md](../observability.md).

## 2. Endpoints

### 2.1 `POST /api/v1/llm/custom-prompt` — synthesize an extraction prompt from a graph model

- **Auth**: `required` (`AuthenticatedUser`).
- **Path params**: none.
- **Query params**: none.
- **Request body** (`application/json`): `CustomPromptGenerationPayloadDTO` (see §4).
  - `graph_model` (required, JSON object): a JSON-Schema-style description of the entities/relationships the caller wants extracted. Free-form `serde_json::Value`. Forwarded into the user prompt template as the `GRAPH_SCHEMA_JSON` placeholder ([`get_llm_router.py:75-80`](https://github.com/topoteretes/cognee/blob/main/cognee/api/v1/llm/routers/get_llm_router.py#L75-L80)). Typical shape (illustrative — the LLM tolerates much variation):
    ```json
    {
      "entity_types": [
        {"name": "Person", "attributes": ["name", "birth_date"]},
        {"name": "Organization", "attributes": ["name", "founded"]}
      ],
      "relationship_types": [
        {"name": "WORKS_AT", "source": "Person", "target": "Organization"}
      ]
    }
    ```
    The HTTP layer does NOT validate the structure beyond "must be a valid JSON object" — Python serializes it via `json.dumps(payload.graph_model)` ([`get_llm_router.py:75`](https://github.com/topoteretes/cognee/blob/main/cognee/api/v1/llm/routers/get_llm_router.py#L75)) and feeds the string into the prompt; any deeper validation is the LLM's problem.
  - `parameters` (optional, JSON object, default `{}`): kwargs forwarded to the LLM adapter. Server-side filtered against `_ALLOWED_LLM_PARAMS = {"temperature", "max_tokens", "top_p", "seed"}` ([`get_llm_router.py:20`](https://github.com/topoteretes/cognee/blob/main/cognee/api/v1/llm/routers/get_llm_router.py#L20)). Anything else is silently dropped (Python: `_safe_params()` at [`get_llm_router.py:23-24`](https://github.com/topoteretes/cognee/blob/main/cognee/api/v1/llm/routers/get_llm_router.py#L23-L24)). Match exactly.
- **Response body** (`200 OK`, `application/json`): `CustomPromptGenerationResponseDTO { custom_prompt: String }`. The prompt is the raw LLM output — a multi-line string suitable for passing as `system_prompt` to `/api/v1/cognify`.
- **Error responses**:

  | Status | Body | Condition |
  |---|---|---|
  | 401 | `{"detail": "Unauthorized"}` | No credential. |
  | 400 | `{"error": "<msg>"}` | `ValueError` raised by the LLM adapter (e.g. malformed schema, prompt-template render failure). Python: [`get_llm_router.py:95-96`](https://github.com/topoteretes/cognee/blob/main/cognee/api/v1/llm/routers/get_llm_router.py#L95-L96). Single-field `{error}` envelope (NOT the canonical `{detail}`). Match exactly. |
  | 409 | `{"error": "<msg>"}` | Catch-all for any other exception (network error, rate limit, OpenAI 5xx). Python: [`get_llm_router.py:97-99`](https://github.com/topoteretes/cognee/blob/main/cognee/api/v1/llm/routers/get_llm_router.py#L97-L99). Logged at `error!` level with the `llm.custom_prompt_generation_failed` event. |
  | 422 | `{"detail": [...], "body": ...}` | DTO deserialization failed (e.g. `graph_model` was not a JSON object, or `parameters` was not an object). Generated by the custom `Json` extractor per [../architecture.md §10](../architecture.md#10-request-validation). |

- **Side effects**: none (no DB writes). The prompt-template render reads two files from the bundled `cognee-cognify` prompt library; that's a read of compiled-in static resources.
- **Delegation target**: `state.lib.llm().acreate_structured_output(LlmRequest { ... }).await` which routes through the configured `Llm` adapter (`OpenAIAdapter`, `LiteRtAdapter`, etc. — see [`crates/llm/`](../../../crates/llm/)). The HTTP handler:
  1. Renders `custom_prompt_generation_user.txt` with `{ "GRAPH_SCHEMA_JSON": serde_json::to_string(&graph_model)? }`.
  2. Renders `custom_prompt_generation_system.txt` with empty context.
  3. Calls `Llm::acreate_structured_output(text_input, system_prompt, response_model=String, **safe_params)`.
  4. Wraps the returned string into `CustomPromptGenerationResponseDTO`.
  
  The two prompt templates currently live in [`cognee/infrastructure/llm/prompts/`](https://github.com/topoteretes/cognee/tree/main/cognee/infrastructure/llm/prompts) on the Python side; the Rust port ships them at `crates/llm/src/prompts/custom_prompt_generation_{user,system}.txt` and renders them via `cognee_llm::prompts::render_prompt`.
- **Validation rules**:
  - `graph_model`: must be a valid JSON object. Empty object `{}` is technically allowed by the type but the LLM output is then unhelpful; we accept and return the LLM's best effort (no special validation).
  - `parameters`: keys outside `_ALLOWED_LLM_PARAMS` are silently dropped. No 400.
- **Rate / size limits**: default (100 MiB body limit). `graph_model` rarely exceeds 50 KiB in practice.
- **Permission gate**: none beyond auth — the LLM call uses the server-configured API key (env `LLM_API_KEY` / `EMBEDDING_API_KEY`), not per-user credentials. Implication: any authenticated user can incur LLM cost. Multi-tenant deployments need a quota layer (out of scope for phase 4 — flag in §6).
- **OpenAPI**: tag `["v1", "llm"]`. Request schema from `CustomPromptGenerationPayloadDTO`; response schema from `CustomPromptGenerationResponseDTO`.
- **Telemetry**: span name `cognee.api.llm.custom_prompt`. Attributes:
  - `cognee.llm.model` — the configured LLM model name (e.g. `"gpt-4o-mini"`).
  - `cognee.llm.provider` — `"openai"` / `"ollama"` / `"litert"`.
  - `cognee.llm.params.keys` — sorted list of parameter keys that survived `_safe_params()` filtering.
  - `cognee.llm.input.tokens` — set after the LLM call returns (when the adapter exposes token counts).
  - `cognee.llm.output.tokens` — same.
  - Python parity: Python emits `send_telemetry("LLM Custom Prompt Endpoint Invoked", ...)` ([`get_llm_router.py:63-72`](https://github.com/topoteretes/cognee/blob/main/cognee/api/v1/llm/routers/get_llm_router.py#L63-L72)) with `parameter_keys` sorted; we emit the same as the `cognee.llm.params.keys` attribute.
- **Python parity notes**:
  - Python's `LLMGateway.acreate_structured_output(..., response_model=str)` ([`get_llm_router.py:87-92`](https://github.com/topoteretes/cognee/blob/main/cognee/api/v1/llm/routers/get_llm_router.py#L87-L92)) abuses the structured-output API by passing `str` as the response model — the LLM is asked to return free-form text. The Rust adapter must support this same pattern; `Llm::acreate_structured_output<String>` should be a valid call (no JSON-schema enforcement when the response type is plain `String`).
  - The HTTP handler does **not** parse the LLM output as JSON — the response is treated as opaque text. Only `infer-schema` parses the LLM output.

### 2.2 `POST /api/v1/llm/infer-schema` — propose a graph model from sample text

- **Auth**: `required`.
- **Path params**: none.
- **Query params**: none.
- **Request body** (`application/json`): `InferSchemaPayloadDTO`.
  - `text` (required, `String`): sample text. Forwarded as the `SAMPLE_TEXT` template variable. No length limit enforced server-side beyond the global body size limit and any LLM context-window limit (the latter produces a 409 with the LLM's own error message). Python: [`get_llm_router.py:39`](https://github.com/topoteretes/cognee/blob/main/cognee/api/v1/llm/routers/get_llm_router.py#L39).
  - `parameters` (optional, JSON object, default `{}`): same `_ALLOWED_LLM_PARAMS` filter as `custom-prompt`.
- **Response body** (`200 OK`, `application/json`): `InferSchemaResponseDTO { graph_schema: serde_json::Value }`. The `graph_schema` is the parsed-and-validated JSON object the LLM produced. Suitable as `graph_model` input to `/custom-prompt` or as the `graph_model` argument to `/api/v1/cognify`.
- **Error responses**:

  | Status | Body | Condition |
  |---|---|---|
  | 401 | `{"detail": "Unauthorized"}` | No credential. |
  | 422 | `{"error": "LLM output is not valid JSON: <serde_json error>"}` | `serde_json::from_str` failed on the LLM output. Python: [`get_llm_router.py:149-153`](https://github.com/topoteretes/cognee/blob/main/cognee/api/v1/llm/routers/get_llm_router.py#L149-L153). The trailing message uses Python's `JSONDecodeError` text format; Rust uses `serde_json::Error::to_string()` which is similar but not byte-identical — flag in §6 for cross-SDK normalization. |
  | 422 | `{"detail": [...], "body": ...}` | DTO deserialization (request body invalid). |
  | 409 | `{"error": "<msg>"}` | Any other exception, including (a) LLM call failure (network, rate limit), (b) `graph_schema_to_graph_model` validation failure on the parsed JSON. Python: [`get_llm_router.py:154-156`](https://github.com/topoteretes/cognee/blob/main/cognee/api/v1/llm/routers/get_llm_router.py#L154-L156). |

- **Side effects**: none (no DB writes).
- **Delegation target**: like `custom-prompt`. The handler:
  1. Renders `infer_schema_user.txt` with `{ "SAMPLE_TEXT": payload.text }`.
  2. Renders `infer_schema_system.txt` with empty context.
  3. Calls `Llm::acreate_structured_output(... response_model=String, ...)`.
  4. Parses the returned string as JSON via `serde_json::from_str::<serde_json::Value>(&output)?`.
  5. Validates by attempting to convert the JSON into the canonical Rust `GraphModel` struct via `cognee_cognify::graph_model::graph_schema_to_graph_model(&parsed)?` (mirrors Python's [`get_llm_router.py:144-146`](https://github.com/topoteretes/cognee/blob/main/cognee/api/v1/llm/routers/get_llm_router.py#L144-L146)). If conversion fails → 409.
  6. Returns the parsed JSON in `graph_schema`.
- **Validation rules**:
  - `text`: max length proposal 1,000,000 chars (Rust safety cap). Python has no explicit limit; LLM context window is the de-facto limit. Document in §6.
  - LLM output: must parse as valid JSON (else 422) and convert to `GraphModel` (else 409). The two-stage validation is intentional in Python — JSON parse error is "user-fixable" (LLM weirdness, retry helps), schema-conversion error is "system-fixable" (the graph model is malformed in a way the LLM can't easily fix without a different prompt).
- **Rate / size limits**: default body limit. Token-window enforcement is the LLM's responsibility.
- **Permission gate**: none beyond auth.
- **OpenAPI**: tag `["v1", "llm"]`. Request schema from `InferSchemaPayloadDTO`; response schema `{graph_schema: object}`.
- **Telemetry**: span name `cognee.api.llm.infer_schema`. Attributes:
  - `cognee.llm.model`, `cognee.llm.provider` — as above.
  - `cognee.llm.input.text_len` — `text.len()` (chars). Python emits `text_length` in `send_telemetry` ([`get_llm_router.py:115`](https://github.com/topoteretes/cognee/blob/main/cognee/api/v1/llm/routers/get_llm_router.py#L115)).
  - `cognee.llm.params.keys`.
  - `cognee.llm.input.tokens`, `cognee.llm.output.tokens` — when available.
- **Python parity notes**:
  - Python re-imports `graph_schema_to_graph_model` inside the handler ([`get_llm_router.py:144`](https://github.com/topoteretes/cognee/blob/main/cognee/api/v1/llm/routers/get_llm_router.py#L144)) to avoid a circular import. Rust just imports it at the top of the file.
  - On JSON-decode failure, Python's error string is `f"LLM output is not valid JSON: {error}"`. Rust must format the same way: `format!("LLM output is not valid JSON: {}", err)`.

## 3. Cross-cutting behavior

- **Both endpoints use a shared `_ALLOWED_LLM_PARAMS` filter** of exactly four keys: `temperature`, `max_tokens`, `top_p`, `seed`. Define the constant once in `crates/http-server/src/routers/llm.rs`:
  ```rust
  const ALLOWED_LLM_PARAMS: &[&str] = &["temperature", "max_tokens", "top_p", "seed"];
  ```
  Apply via a `safe_params(serde_json::Value) -> serde_json::Value` helper. Cross-SDK note: Python uses a `set` for the lookup; Rust uses a slice (4 items, linear search is fine).
- **Both endpoints use the canonical `Llm::acreate_structured_output` with `String` as the response type.** This is a no-op for structured-output enforcement — the LLM is asked to return free-form text. The `infer-schema` handler does its own JSON parse on top.
- **Error envelope is `{error}` (single field), not `{detail}`.** Match Python exactly. Use the dedicated `ApiError::LlmError(StatusCode, String)` variant.
- **No streaming.** Both endpoints are blocking; the LLM call typically takes 5–30s. The frontend is expected to show a spinner. Streaming the LLM output is out of scope for phase 4 — flag in §6.

## 4. DTO definitions

```rust
use serde::{Deserialize, Serialize};
use serde_json::Value;
use utoipa::ToSchema;

/// Mirrors Python `CustomPromptGenerationPayloadDTO` (`get_llm_router.py:27-32`).
#[derive(Debug, Clone, Deserialize, Serialize, ToSchema)]
#[serde(deny_unknown_fields)]
pub struct CustomPromptGenerationPayloadDTO {
    /// Free-form JSON object describing the desired graph model. Forwarded
    /// as the `GRAPH_SCHEMA_JSON` template variable. The Rust HTTP layer
    /// does NOT validate the structure — the LLM tolerates variation.
    /// Recommended shape:
    ///   ```json
    ///   {"entity_types": [...], "relationship_types": [...]}
    ///   ```
    pub graph_model: Value,

    /// Kwargs forwarded to the LLM adapter. Filtered server-side against
    /// `ALLOWED_LLM_PARAMS = ["temperature", "max_tokens", "top_p", "seed"]`.
    /// Other keys are silently dropped (Python `_safe_params()` parity).
    #[serde(default)]
    pub parameters: Value,
}

#[derive(Debug, Clone, Serialize, ToSchema)]
pub struct CustomPromptGenerationResponseDTO {
    pub custom_prompt: String,
}

/// Mirrors Python `InferSchemaPayloadDTO` (`get_llm_router.py:39-44`).
#[derive(Debug, Clone, Deserialize, Serialize, ToSchema)]
#[serde(deny_unknown_fields)]
pub struct InferSchemaPayloadDTO {
    /// Sample text to analyze for entity types and relationships.
    pub text: String,

    /// Same filter rules as `CustomPromptGenerationPayloadDTO::parameters`.
    #[serde(default)]
    pub parameters: Value,
}

#[derive(Debug, Clone, Serialize, ToSchema)]
pub struct InferSchemaResponseDTO {
    /// Parsed-and-validated JSON object the LLM produced. Suitable as the
    /// `graph_model` argument to `/v1/llm/custom-prompt` or `/v1/cognify`.
    pub graph_schema: Value,
}
```

### `graph_model` shape — what the frontend actually posts

Based on the Python `graph_schema_to_graph_model` validator ([`cognee/shared/graph_model_utils.py`](https://github.com/topoteretes/cognee/blob/main/cognee/shared/graph_model_utils.py)) and the prompt templates, the canonical (but not enforced) shape is:

```json
{
  "name": "MyDomainModel",
  "entity_types": [
    {
      "name": "Person",
      "description": "An individual human",
      "attributes": [
        {"name": "name", "type": "string"},
        {"name": "birth_date", "type": "string", "format": "date"}
      ]
    }
  ],
  "relationship_types": [
    {
      "name": "WORKS_AT",
      "source_type": "Person",
      "target_type": "Organization",
      "description": "Employment relationship"
    }
  ]
}
```

The Rust DTO declares it as `serde_json::Value` to keep the HTTP layer permissive — the LLM, not the HTTP layer, is the validator. `graph_schema_to_graph_model` is the only point of enforcement (in `infer-schema`).

## 5. Implementation tasks

1. Add `crates/http-server/src/dto/llm.rs` with the four DTOs above and a `safe_params(Value) -> Value` helper.
2. Port the four prompt templates (`custom_prompt_generation_{user,system}.txt`, `infer_schema_{user,system}.txt`) from Python to `crates/llm/src/prompts/`. Wire `cognee_llm::prompts::render_prompt(name, ctx)` to load + Jinja-style render. (If `cognee-llm` already has a prompt loader, reuse it; otherwise add one.)
3. Add `crates/http-server/src/routers/llm.rs` with `post_custom_prompt`, `post_infer_schema`, and `pub fn router()` that mounts both POSTs at `/custom-prompt` and `/infer-schema` respectively.
4. Wire `nest("/llm", llm::router())` in `build_router()`.
5. Extend `crates/http-server/src/error.rs` with `ApiError::LlmError(StatusCode, String)` returning `{error}` envelope.
6. Ensure `cognee_cognify::graph_model::graph_schema_to_graph_model` exists and returns a clean error type. If not, port from Python.
7. Unit tests: `safe_params` filters correctly; DTO deserialization with each ALLOWED key; DTO with disallowed keys (silently dropped, no error); `infer-schema` JSON-decode error path produces `422 {"error": "LLM output is not valid JSON: ..."}`.
8. Integration tests in `crates/http-server/tests/test_llm.rs`:
   - With `MOCK_LLM=true` (returns canned responses), POST `/custom-prompt` → 200 with the canned prompt.
   - POST `/infer-schema` with a mock returning valid JSON → 200; with invalid JSON → 422; with valid JSON that fails `graph_schema_to_graph_model` → 409.
   - POST `/custom-prompt` with no auth → 401.
9. Cross-SDK parity test in `e2e-cross-sdk/harness/test_http_llm.py`: with the same OpenAI-compatible mock backend, both Python uvicorn and Rust binary produce identical response shapes (modulo LLM nondeterminism — pin `seed=42`).

## 6. Open questions

1. **Per-user LLM cost quotas**: any authenticated user can trigger paid LLM calls. Add a quota layer at the HTTP middleware level? Out of scope for phase 4; flag for `tenants.md` or a new `quotas.md`.
2. **`text` length cap**: Python relies on the LLM context window. We propose 1,000,000 chars in Rust as a defensive cap. Confirm or adjust.
3. **Streaming `custom-prompt`**: returning the prompt token-by-token via SSE would be a UX win for the frontend. Out of scope for phase 4.
4. **Mock LLM mode for testing**: the `MOCK_LLM` / `MOCK_EMBEDDING` env vars exist for embeddings but not LLM. Add `MOCK_LLM=true` that returns canned prompt-generation / schema responses for CI? Recommend yes — same pattern as `MockEmbeddingEngine`.
5. **JSON-decode error message format**: Python's `JSONDecodeError` message format differs slightly from `serde_json::Error`. For cross-SDK parity tests, normalize the error message to `"LLM output is not valid JSON: <reason>"` where `<reason>` matches a regex (not a literal string). Document the regex in the parity test.
