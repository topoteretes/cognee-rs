## Rationale

Per [decision 1](../01-otel-otlp-export.md#design-decisions-locked) the
`telemetry` cargo feature is **off** by default in `cognee-lib`,
`cognee-cli`, and `cognee-http-server`. The implication for CI is
twofold and asymmetric:

1. **The existing `lib-tests` job in
   [`.github/workflows/ci.yml`](../../../.github/workflows/ci.yml)
   already exercises the noop fallback path** ŌĆö `cargo check
   --all-targets` and `bash scripts/run_tests_with_openai.sh` both run
   without `--features telemetry`, so they hit the
   `#[cfg(not(feature = "telemetry"))]` arm of `init_telemetry`
   (introduced in [task 08](08-noop-fallback.md)). This is good: it
   means a build that does not opt in keeps working.

2. **No CI lane currently exercises the `telemetry`-on path.** Without
   a dedicated lane, breakage in the OTEL-gated code (mismatched
   API after a `tracing-opentelemetry` minor bump, missing `cfg`
   guard, broken `init_telemetry` signature) ships green ŌĆö the failure
   only surfaces when a downstream user runs `cargo install
   cognee-cli --features telemetry` or `cargo install cognee-lib
   --features telemetry`. That is exactly the silent-breakage
   class of bug the project guide warns about under "Feature
   strategy".

The remediation is to add three new lanes:

- A **fast `cargo check --features telemetry`** lane to catch
  compile-time breakage early (under a minute on a warm cache,
  cheaper than a full test run).
- A **`cargo test -p cognee-observability --features telemetry`**
  lane to run the unit tests from
  [task 09](09-observability-unit-tests.md) and the integration test from
  [task 10](10-otel-export-integration-test.md). Scoping to `-p
  cognee-observability` keeps this lane focused: the rest of the
  workspace's tests are already covered by the existing test job
  (which now also rebuilds the dependent crates with the feature on
  via the `cognee-lib`/`cognee-cli` propagation from
  [task 03](03-cognee-lib-feature-wiring.md)).
- An **explicit `cargo check --no-default-features` lane** for
  `cognee-lib` to catch `cfg`-gated noop drift (a common Rust footgun:
  add a public function inside `#[cfg(feature = "telemetry")]` and
  forget to mirror its signature inside `#[cfg(not(feature = "telemetry"))]`).
  The existing `lib-tests` job *runs* with the feature off, but it
  builds with default features that may grow over time and accidentally
  start enabling `telemetry` transitively; the no-default-features lane
  is the canary for that drift.

A fourth, **optional** lane runs `cargo doc --features telemetry` to
catch rustdoc intra-doc-link breakage from
[task 11](11-user-facing-documentation.md). The existing CI already runs `cargo
doc --no-deps` on the default-features build, so this only adds
coverage for `#[cfg(feature = "telemetry")]` `///` comments (which
the docless build skips entirely).

### 2. Modify `ci.yml` `lint` job ŌĆö add `--features telemetry` check + clippy

Add two steps after the existing `clippy` step. The
`--features telemetry` clippy run is gated against a Rust matrix of
one for now (we keep the lane simple; future expansion can promote
this to a matrix). We also add the no-default-features check here
because it is a fast compile-only step that fits the lint job's
budget.

Diff (apply to
[`.github/workflows/ci.yml`](../../../.github/workflows/ci.yml)):

```diff
       - name: rustfmt (check)
         run: cargo fmt --all -- --check

       - name: Compilation check
         run: cargo check --all-targets

       - name: clippy
         run: cargo clippy --all-targets --all-features -- -D warnings

+      # Telemetry feature is OFF by default per design decision 1, so the
+      # default `cargo check --all-targets` above only covers the noop arm.
+      # This step exercises the OTEL-on code path (init_telemetry, OTLP
+      # exporter, tracing-opentelemetry bridge) at compile time.
+      - name: Compilation check (telemetry feature)
+        run: cargo check --all-targets --features telemetry
+
+      # Mirror of the above for clippy. Catches cfg-gated lints that the
+      # noop-arm clippy run cannot see. We do NOT pass `--all-features`
+      # because the workspace also ships android-* features that change
+      # link targets; we want the precise (default + telemetry) combo a
+      # downstream user would enable via `cargo build --features telemetry`.
+      - name: clippy (telemetry feature)
+        run: cargo clippy --all-targets --features telemetry -- -D warnings
+
+      # Canary for cfg(not(feature = "telemetry")) drift: ensures the noop
+      # path in cognee-observability compiles cleanly even if `default`
+      # ever grows a transitive dep that accidentally enables telemetry.
+      # Scoped to `-p cognee-lib` to keep the build small (no rebuild of
+      # the full workspace's binaries).
+      - name: Compilation check (no default features)
+        run: cargo check -p cognee-lib --no-default-features
```

Note on `--all-features` versus `--features telemetry`: the existing
clippy step already uses `--all-features`, which technically already
covers `telemetry`. We keep it AND add an explicit telemetry-only
lane so that:
- the failure signal is unambiguous (the matrix entry name says
  *which* feature broke);
- when a future feature lands that conflicts with `telemetry` under
  `--all-features`, the explicit lane still passes and pins the
  blame on the new feature, not on `telemetry`.

### 3. Modify `ci.yml` `test` job ŌĆö add a telemetry test lane

Add a new step after the existing `Test` step. Scoped to
`cognee-observability` to keep run time low ŌĆö the rest of the
workspace is already tested under the noop arm by the existing step,
and the unit/integration tests for OTEL all live in
`cognee-observability` per
[task 02](02-observability-crate-scaffold.md) and
[task 10](10-otel-export-integration-test.md).

Diff (apply to
[`.github/workflows/ci.yml`](../../../.github/workflows/ci.yml)
inside the `test` job):

```diff
       - name: Test
         run: bash scripts/run_tests_with_openai.sh

+      # Telemetry-feature test lane. The integration test from task 01-10
+      # spawns an in-process tonic gRPC server on 127.0.0.1:0; no external
+      # OTLP collector is needed. Scoped to `-p cognee-observability`
+      # because that is where init_telemetry, OtelGuard, and the OTLP
+      # parity tests live; the rest of the workspace is already tested
+      # in the noop arm above.
+      - name: Test (telemetry feature)
+        run: cargo test -p cognee-observability --features telemetry -- --nocapture
+
       - name: Build docs
         run: cargo doc --no-deps
+
+      # Optional: rustdoc with the telemetry feature on, to catch
+      # cfg(feature = "telemetry") doc-link breakage from task 01-11.
+      - name: Build docs (telemetry feature)
+        run: cargo doc --no-deps --features telemetry
```

The `--nocapture` flag matches `scripts/run_tests_with_openai.sh`'s
behaviour (the existing tests already print useful diagnostics to
stdout) and helps debug OTLP exporter handshakes when the lane fails.

We deliberately **do not** add `--test-threads=1` here. The existing
script forces single-threaded execution because LLM-backed tests
serialise on the OpenAI rate limit; the telemetry tests do not call
LLMs and parallel execution shaves real time off CI.

### 4. Cache key handling

`Swatinem/rust-cache@v2` keys on Cargo.lock and the rustc fingerprint
by default; it does **not** automatically segregate caches by feature
flags. Without intervention, the new `--features telemetry` step will
warm the same cache as the default-feature step and cause cache
churn (each job invalidates the other's `target/`).

Fix: add a `key:` suffix to differentiate the two feature states. The
existing `lint` and `test` jobs share `shared-key: workspace-v3` (a
single key for the whole workflow); we widen this to a
feature-aware key.

Diff:

```diff
       - name: Cache cargo/target
         uses: Swatinem/rust-cache@v2
         with:
-          shared-key: workspace-v3
+          shared-key: workspace-v3
+          # Append a suffix that includes the feature combo this job
+          # builds with, so the telemetry-on and telemetry-off lanes do
+          # not evict each other.
+          key: ${{ matrix.features || 'default' }}
```

If we do not introduce a `matrix.features` variable (the simplest
implementation just adds inline steps without a matrix), use a
literal suffix per step instead by setting the env var
`CARGO_TARGET_DIR` to a per-feature subdirectory:

```diff
+      - name: Compilation check (telemetry feature)
+        env:
+          CARGO_TARGET_DIR: target/telemetry
+        run: cargo check --all-targets --features telemetry
```

Pick **one** approach (`matrix.features` is cleaner; the
`CARGO_TARGET_DIR` workaround keeps the diff small). This sub-doc
recommends the `CARGO_TARGET_DIR` form for the initial PR ŌĆö it
avoids restructuring the job and the cache size cost is bounded by
the runner's free-disk-space step.

## Verification

There is no good way to dry-run a GitHub Actions workflow without
either `act` (which has well-known limitations with
`Swatinem/rust-cache@v2`) or pushing to a branch. The recommended
local-test approach has three rungs:

1. **Run the new commands locally and confirm they pass** ŌĆö
   `cargo check --all-targets --features telemetry`,
   `cargo clippy --all-targets --features telemetry -- -D warnings`,
   `cargo check -p cognee-lib --no-default-features`,
   `cargo test -p cognee-observability --features telemetry`. This
   catches every kind of error the lane catches except for
   GitHub-Actions-specific issues (cache key collisions, runner
   environment differences).

2. **Run `scripts/check_all.sh`** ŌĆö exercises the appended steps in
   the same order CI does, against the same workspace.

3. **Push to a feature branch with intentional breakage** to confirm
   CI catches it. Two suggested experiments:
   - Comment out a `#[cfg(feature = "telemetry")]` guard in
     `crates/observability/src/otel.rs` so the file references an
     OTEL symbol from the noop arm ŌĆö confirm the
     `Compilation check (no default features)` lane goes red.
   - Add a `compile_error!()` inside an
     `#[cfg(feature = "telemetry")]` block ŌĆö confirm the
     `Compilation check (telemetry feature)` lane goes red while the
     default `Compilation check` lane stays green.

   Delete both branches once verified ŌĆö these are throwaway PRs that
   exist only to prove the new lanes wired up correctly.

## Risks

- **Increased CI time.** Each new lane roughly doubles the cargo
  build cost for the affected job (a fresh telemetry-on build
  recompiles `opentelemetry`, `opentelemetry_sdk`,
  `opentelemetry-otlp`, `tonic`, and their transitive
  dependencies ŌĆö non-trivial). Mitigations:
  - Re-use `Swatinem/rust-cache@v2` per
    [┬¦ 4](#4-cache-key-handling) so subsequent runs hit the cache.
  - Scope `Test (telemetry feature)` to `-p cognee-observability` (one
    crate) rather than the full workspace.
  - The `Compilation check (no default features)` lane is scoped to
    `-p cognee-lib` and is therefore cheap.
- **Cache pollution.** Running `cargo` with two feature combinations
  against the same `target/` directory triggers fingerprint
  invalidation on every step swap, even though the *artifact* set on
  disk is correct. The `CARGO_TARGET_DIR=target/telemetry` per-step
  override (and the corresponding `Swatinem/rust-cache` key suffix
  if we adopt the matrix form later) is the standard fix.
- **Network-socket use in CI.** The integration test from
  [task 10](10-otel-export-integration-test.md) binds `127.0.0.1:0` and dials
  itself. GitHub-hosted runners have no firewall on the loopback
  interface and the OS picks a free ephemeral port, so this is a
  no-op risk. Document it explicitly in the test's module-level
  doc-comment for future maintainers.
- **`--all-features` already covers `telemetry`.** The current
  clippy step uses `--all-features`, so technically the explicit
  `--features telemetry` lane duplicates coverage. The duplication is
  intentional: `--all-features` enables every feature flag in the
  workspace simultaneously, including platform-specific ones and
  combinations no real user would build, so a failure under
  `--all-features` does not pin blame on `telemetry`. Keeping the
  explicit lane gives an unambiguous failure signal.
- **Future feature drift.** If a follow-up task adds another
  optional feature (e.g. `metrics`, `langfuse`, etc. from
  [decision 12](../01-otel-otlp-export.md#design-decisions-locked)),
  the same lane-doubling pattern needs to repeat. Promote the inline
  steps to a `strategy.matrix.features:` array at that point.

## Open / clarifying questions

- Should we add a `cargo deny` lane to catch licence and CVE drift
  in the new OTEL deps? `opentelemetry` and friends are Apache-2.0,
  which is workspace-compatible, but `tonic` pulls in a sizeable
  transitive tree. **Recommendation: yes**, in a follow-up PR. Out of
  scope for this gap. Track as a follow-up issue once
  [task 01](01-workspace-otel-deps.md) lands and the lockfile is
  visible.
- Should the `Test (telemetry feature)` lane run on macOS / Windows?
  The current `ci.yml` is Linux-only (`ubuntu-latest` for every job),
  and the OTLP tonic exporter is portable, but cross-platform
  hardening is out of scope for this gap. Track as a follow-up.
- Should we adopt a true matrix (`strategy.matrix.features:
  ['', 'telemetry']`) to halve the maintenance cost of duplicated
  steps? **Recommendation: not in the initial PR.** Defer until a
  third feature (per the previous question) actually arrives ŌĆö
  premature matrixification adds complexity without payoff while we
  have only two states.
